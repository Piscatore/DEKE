# OP-008d: Deke.Api fails fast when ApiKey is unconfigured

## Status: done (2026-07-14)

## Contract

- **Goal:** Make `Deke.Api` refuse to start (or hard-fail early, before
  serving requests) when configuration key `"ApiKey"` is unconfigured, in
  every environment including local dev, per ADR-0008's resolution.
- **Context budget:**
  - ALWAYS: `docs/GLOSSARY.md`, `overhaul/STATE.md`
  - THIS PACKET: `docs/adr/ADR-0008-apikey-auth-handler-noresult-blocks-unconfigured-dev-auth.md`
    (full, including Resolution — the decision this packet implements);
    `src/Deke.Api/Auth/ApiKeyAuthHandler.cs` (full — the `NoResult()` branch
    being replaced by a startup check, not a runtime one); `src/Deke.Api/Program.cs`
    (full — where the startup check belongs, likely right after
    `builder.Build()` or during `builder.Services.AddAuthentication` setup);
    `src/Deke.Api/appsettings.json` (the `"ApiKey": ""` default); top-level
    `CLAUDE.md` (the quick-start curl examples that currently omit
    `X-Api-Key`, to be corrected).
  - Known ties to prior packets: `docs/PROJECT-MAP.md`'s "API Key
    Authentication" entry (OP-004c) — read it for the exact failure mode
    being fixed.
- **Tool budget:** This packet needs a **code-capable agent/session** — Read/
  Edit against `src/Deke.Api/Program.cs`, `src/Deke.Api/Auth/ApiKeyAuthHandler.cs`,
  and top-level `CLAUDE.md`; `dotnet build`/`dotnet run` to confirm the
  fail-fast behavior actually triggers with an empty/missing key and actually
  starts cleanly with one configured.
- **Recommended tier:** Sonnet-class, default thinking — small, well-scoped
  code fix per `docs/TOOLING.md`'s model/thinking tiers table, not a design
  packet (the design decision is already made in ADR-0008).
- **Inputs:** ADR-0008 (accepted, resolution recorded 2026-07-07); OP-004c
  (Deke.Api mapping sweep, complete — original finding).
- **Outputs:** `Program.cs` (or a small new startup-validation helper) throws/
  exits with a clear error message identifying the missing `"ApiKey"`
  configuration key before the app starts serving requests;
  `ApiKeyAuthHandler.cs`'s `HandleAuthenticateAsync` unconfigured-key branch
  removed or simplified (no longer reachable in normal operation once
  startup enforces the key exists — decide whether to leave a defensive
  fallback or delete the branch entirely, per whichever reads cleaner);
  top-level `CLAUDE.md`'s Quick Start / "Add a source to monitor" /
  "Search facts" curl examples updated to set an `ApiKey` value (e.g. via
  `appsettings.Development.local.json`) and include the `X-Api-Key` header
  where required; `docs/PROJECT-MAP.md`'s "API Key Authentication" entry
  updated to remove the ADR-0008 escalation note and record the fix;
  `overhaul/STATE.md` (new top entry).
- **Done when:** `dotnet run --project src/Deke.Api` with no `"ApiKey"`
  configured fails fast with a clear, actionable error instead of starting;
  with `"ApiKey"` configured, all endpoints behave as documented in
  `specification.md:655`; `CLAUDE.md`'s examples work as written against a
  freshly configured instance; `STATE.md` has a new top entry.
- **Human touchpoints:** none expected — this packet implements an already-
  accepted ADR with a fully specified decision.
- **Out of scope:** ADR-0007's `Llm/` retirement (OP-008c); the
  implementation-behind-spec gap logged in `PARKING-LOT.md` (missing
  `PUT`/`DELETE` fact and source endpoints) — separate, not part of this fix;
  any change to how the API key itself is validated once present (constant-
  time comparison, rotation, etc.) beyond what `ApiKeyAuthHandler.cs` already
  does.

   the ADR's `Consequences` field defines the new packet(s).
